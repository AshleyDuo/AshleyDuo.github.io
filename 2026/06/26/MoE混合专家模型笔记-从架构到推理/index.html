一、MoE 模型的性能分析与工作机制

🎯 问题

多专家大模型在问题处理能力上会比普通的大语言模型强多少？既然它学习了很多我不需要的领域的知识，在使用的时候为什么有差距？

回答

MoE 的真正价值在于用更经济的计算成本撬动更大的模型容量，而不是绝对更强的性能。它在简单任务上效率极高，在复杂推理上甚至可能弱于稠密模型。

定量分析

这种“能力与成本的权衡”，在不同任务上表现为各具特色的性能差距：

对比维度 MoE 模型优势 普通模型（稠密）优势
简单生成任务 效率极高
推理速度可提升数倍，质量损失小于8%，吞吐量是同类稠密模型的2.3倍。		 差距不明显
但同等算力下，速度仅为MoE的五分之一。

性能提升主要依赖粗暴的参数堆砌。

典型案例是Mixtral 8x7B，它以约120亿的激活参数量，在很多任务上达到了Llama 2 70B的效果，而后者每次推理要激活全部700亿参数。

MoE 工作机制

MoE模型就像一个由“门控网络（Router/Gate）”和成百上千个“专家”构成的机构，它高效且省力，但有时也略显“沟通不畅”。这里的“专家”并非像医学、法律、编程那样基于人类定义的、泾渭分明的知识模块，其专长更多是由数据自身的几何结构在训练中自然形成，而非来自刻意的知识领域划分。

  • 门控网络：聪明的“项目经理”：它快速分析任务，将工作分配给最合适的少数专家处理，避免全员出动，从而提升效率。
  • 专家：“偏科”但高效的执行者：每个专家模型只在自己擅长的特定领域高效工作，但对全局知识的统筹能力较弱。
  • 知识冗余（Knowledge Redundancy）：隐秘的效率杀手：由于任务分配机制的高度统一，多个专家在训练中可能学到非常相似的知识模式，造成重复存储和计算的浪费。这也直接回答了你的问题——它内置的大量领域知识，事实上处于相当程度的冗余状态，远未实现理想中的高度专业化分工。

你提出的这个问题非常关键，它揭示了这种“聪明”架构的代价：能力优势与知识冗余之间的落差，恰恰是MoE模型为“高效”所付出的核心代价。

  • 专家协作的沟通成本：复杂推理需要全局统筹，但MoE架构在处理时需频繁在专家间路由决策，容易产生“推理断裂”，如同一个临时拼凑的项目组，沟通成本高昂。
  • 知识“孤岛”的天然限制：每个专家只关心自己的“一亩三分地”，缺乏对整个任务的整体理解。当问题需要跨领域知识融合时，模型就显得捉襟见肘。
  • 知识的冗余与浪费：大量专家功能重叠，导致宝贵的模型容量被浪费，无法真正发挥“1+1>2”的效果。

所以，回答你的问题：

  1. 强多少？ 不能简单量化，它是在效率和效果之间寻找平衡。在简单任务上，它可能“强”数倍；在复杂推理上，甚至可能“弱”23%。
  2. 为什么有差距？ 因为它并非真正的“精英团队”，而更像一个聪明的“项目经理”带领一群“偏科”但高效、甚至可能有些（知识）冗余的员工。它在日常任务中效率极高，但在需要深度、全局思考的复杂问题上，沟通成本和知识孤岛就暴露了短板。

了解了MoE的特性后，想不想知道你的具体任务场景更适合选择MoE模型还是普通的稠密模型吗？我可以帮你进一步分析～

📝 知识点小结

  1. MoE 追求的是效率与效果的平衡，不是绝对更强的性能。
  2. 简单任务上 MoE 效率极高（数倍优势）；复杂推理上甚至可能弱 23%。
  3. 知识冗余和专家沟通成本是 MoE”高效”架构的核心代价。
  4. 典型案例：Mixtral 8x7B 以 120 亿激活参数达到 Llama 2 70B 的效果。

二、为什么普通模型不能选择性激活？

🎯 问题

既然 MoE 可以只激活部分专家，为什么普通稠密模型不能也”选择性激活”部分参数？

回答

稠密模型不是不想选择性激活，而是不能——其全连接参数结构和密集梯度训练方式使得每个参数都深度纠缠在全局计算中。

论证拆解

① 参数结构：全连接 vs. 模块化

稠密模型的每一层都是全连接网络（Fully-Connected Network），每个参数都对最终结果产生难以分割的全局性影响。MoE 将 FFN 层替换为多个独立的专家，天然是松耦合的模块化结构。

② 训练方式：密集梯度 vs. 稀疏梯度

稠密模型的每个训练样本产生关于几乎所有参数的梯度（密集梯度，Dense Gradient），所有参数必须时刻”待命”。MoE 每个样本只激活少数专家，反向传播时只有被激活的专家收到梯度（稀疏梯度，Sparse Gradient）。

③ 硬件与计算效率

现代 GPU 极度优化规则的大型密集矩阵乘法。强行让稠密模型跳过某些神经元，矩阵中会出现大量零值，导致计算效率崩溃。

🔄 能否改造？

技术上可以，但那本质上就是把稠密模型转变为 MoE 架构。需要注意：静态剪枝（Static Pruning）（一次性移除不重要参数）≠ 动态选择性激活（每次推理根据输入改变激活的神经元）。

📝 知识点小结

  1. 稠密模型的参数是深度纠缠的，无法独立调用。
  2. 稠密模型的密集梯度训练让所有参数必须时刻待命。
  3. 想改造稠密模型实现选择性激活，本质就是把它变成 MoE 架构。
  4. 静态剪枝 ≠ 动态选择性激活，两者是不同层面的优化。

三、MoE 的参数量为什么越来越多？

🎯 问题

为什么 MoE 的参数量一直在膨胀？从几百亿到几千亿，甚至上万亿？

回答

MoE 参数膨胀的核心逻辑是：总参数量（知识储备）与激活参数量（推理成本）解耦，增加参数主要增加存储成本而非计算成本。

MoE 的核心特性——总参数量（知识储备）与激活参数量（单次算力）脱钩——催生了一个新公式：总参数增量 - 成本微增 ≈ 净收益。

成本结构对比

维度 稠密模型 MoE 模型
核心目标 性能随参数同步增长 知识容量扩展，算力可控
成本结构 总参数 ≈ 激活参数，算力线性增长 总参数 ≫ 激活参数，算力相对固定
参数增长的代价 算力成本飙升 主要是存储成本增加
根本限制 算力与存储强耦合 算力与存储解耦，天花板高得多

三大驱动力

1️⃣ 极致的经济账：增加总参数只需增加存储成本，推理计算成本几乎不变。

2️⃣ 全新的规模定律（Scaling Law）：DeepMind 等研究发现，MoE 遵循与稠密模型不同的规模定律——总参数越大，激活参数比例应越稀疏。

3️⃣ 实证：DeepSeek 的参数跃迁

模型 总参数 激活参数 激活比例
DeepSeek-V2 236B 37B 15.7%
DeepSeek-V3 671B 37B 5.5%
DeepSeek-V4 1.6T 49B ~3%

从 V3 到 V4，总参数增长约 2.4 倍，激活参数仅增长 32%。

🧠 唯一约束：从”算力墙”到”存储墙”

当总参数量膨胀到万亿级别，瓶颈就从”算力不够算”变成了”显存放不下”。谷歌 DeepMind 的研究已成功将专家数量扩展至百万级别，同时保持计算成本不变。

📝 知识点小结

  1. MoE 参数膨胀的核心逻辑：知识容量 ≈ 总参数，推理成本 ≈ 激活参数，两者解耦。
  2. DeepSeek 系列是从 236B → 1.6T+ 的实证。
  3. 瓶颈从”算力墙”变为”存储墙”，万亿级参数需要海量显存。

四、部署时能否删除专家来减小显存？

🎯 问题

为什么不能在部署时直接删除某些专家？即使把输出置零也不行吗？

回答

不能直接删除专家，因为门控网络的权重与专家输出幅值深度耦合。置零输出虽然不会让模型崩溃，但会导致性能骤降。

论证拆解

① 为什么不能直接删除专家？

门控网络的权重是在全体专家上训练的。直接删除某专家会带来路由失灵（门控仍试图分配任务给已删除的专家）和动态需求问题。GPU 对规则的密集矩阵乘法极度优化，删除专家会导致内存排列不规则，严重拖慢计算速度。

以 4 个专家（A、B、C、D）为例，输入”苹果”：

  • 门控打分：A=0.5, B=10.0, C=6.0, D=3.0
  • 归一化权重：[0.001, 0.76, 0.20, 0.039]
  • 选中专家 B（权重 0.76）和 C（权重 0.20）

强行删除专家 B → 门控只能选 C 和 D，输出完全偏离预期。

② 为什么没有”超级替补”？

专家能力是非对称的。专家 B 的高权重源于它捕捉了输入中的特定模式，专家 C 可能擅长完全不同的事。门控的全部训练是为了”找到最优”，没有学习”如果 X 不行就用 Y 代替”的备选逻辑。

③ 把专家输出置零可行吗？

这是一个聪明且有工程价值的想法（”掩码”，Masking），但依然会显著损害性能。

数值示例：

  • 正常输出 = 0.001×2 + 0.76×10 + 0.20×6 + 0.039×1 = 8.841
  • 专家 B 置零后 = 0.001×2 + 0.76×0 + 0.20×6 + 0.039×1 = 1.241
  • 从 8.8 暴跌到 1.2，模型输出完全偏离预期。

原因：门控网络不是在”分类”，而是在做联合预测——它预测的是每个专家应贡献的数值量级，而不仅仅是哪个专家更相关。门控权重与专家输出幅值是高度耦合的。

🛠️ 正确的专家”瘦身”方法

路径一：静态剪枝（Static Pruning）——永久合并冗余专家

  • 效果：在 Mixtral 8x7B 上删除 75% 的专家，性能几乎不变；推理速度提升 6 倍
  • 典型方法：EEP、DERN、SHAPE、PreMoe
  • 缺点：不可逆的永久删除，可能导致性能下降

路径二：动态加载（On-demand Loading）——不删专家，只删显存占用

  • 典型系统：fMoE（延迟降低 47%），华为昇腾（显存减半），OD-MoE（<1GB 显存运行）
  • 优点：完全无损，部署灵活
  • 缺点：引入 PCIe 传输延迟，需要预取、缓存、调度策略

📝 知识点小结

  1. 门控网络不是分类器——它学习的是权重-专家贡献的联合模式。删除专家会破坏这种耦合。
  2. 直接置零输出不会让模型崩溃，但会导致性能骤降（8.8 → 1.2）。
  3. 静态剪枝是永久性瘦身，动态加载是让模型做”仰卧起坐”。

五、训练门控网络需要整个模型配合

🎯 问题

训练门控网络是否必须整个模型配合？如果专家已经很好了，为什么不能只训练门控？

回答

门控网络不是独立分类器，必须与专家联合训练。冻结专家只训练门控会导致性能下降约 20%。

论证拆解

① 为什么必须联合训练？

门控网络没有独立的”正确路由标签”，唯一监督信号是最终任务损失。门控和专家必须协同进化。负载均衡需要全局约束（辅助损失，Auxiliary Loss，防止某些专家”饿死”）。梯度通过”权重 × 专家输出”路径传播，专家的输出必须参与计算。

② 冻结专家、只训练门控会怎样？

设置 验证损失 备注
联合训练（基准） 1.85 专家和门控同时更新
冻结专家，只训练门控 2.23 性能下降约 20%
只训练专家，冻结门控 2.41 更差

原因：门控学习的是专家输出的数值模式，新门控从零开始不知道每个专家输出的”量级”，需要几千步重新发现。

③ 什么时候”只训练门控”有意义？

  • 任务增量学习：学会新任务但不想改变原有专家知识
  • 领域适应（Domain Adaptation）：源领域和目标领域分布不同
  • 硬件受限的剪枝：删除某些专家后，对剩余专家轻量微调

📝 知识点小结

  1. 门控网络不是独立的分类器，它是整个模型中普通的一层。
  2. 能不能只训练门控？能。但不推荐，因为性能下降约 20%。
  3. 专家的”优良”是在与旧门控的协同中定义的，换新门控会导致数值匹配错乱。

六、门控网络的本质与专家的真实输入

🎯 问题

门控网络的输入输出本质是什么？专家到底有几个输入？MoE 的条件控制机制怎么理解？

回答

门控网络的输出是路由权重向量，不包含输入文本。专家只有一个输入——上下文向量。MoE 在网络内部嵌入了显式的路由控制器。

核心要点

  • 门控的输出不包含输入文本向量：它接收输入向量，输出一个关于专家的权重分布。

    • 输入：隐藏状态向量 x（形状 [d_model]）
    • 处理：logits = x · W_router（形状 [num_experts]）
    • 输出：权重向量 [w1, w2, ..., wE]（路由决策向量，不是输入向量本身）

  • 门控是大模型内部的一个子模块：它是 MoE 层的一部分，不是一条独立的并行路径。门控参数和专家参数通过同一个损失函数更新。

  • 专家只有一个输入——上下文向量：门控并不作为第二个输入提供给专家。门控通过外部控制（决定调用哪个专家、如何加权组合）起作用，但专家的内部计算完全独立于门控信号。

  • 不激活的专家完全不会被计算：门控先决定索引，然后只对被选中的专家调用前向传播。其余专家在此次前向传播中完全不存在于计算图中（仅占用显存）。

组件输入输出速查

组件 输入 输出
门控网络 上下文向量 x 权重向量 w（长度为专家数）
每个专家 上下文向量 x（同一个） 变换后的向量 y_i
MoE 层最终 (从上述组合而来) 加权和 ∑ w_i × y_i

MoE 的条件控制机制

MoE 在网络内部嵌入了一个显式的路由控制器，这个控制器根据输入动态选择只激活一小部分专家。它跳出了普通稠密网络”所有参数都要参与计算”的固定模式，但又依然是整个神经网络的可训练组成部分。这是模型自己学会的稀疏计算模式（条件计算，Conditional Computation）。

📝 知识点小结

  1. 门控网络的输出是路由权重向量，不包含输入文本。
  2. 专家唯一输入就是上下文向量，门控不提供第二个输入。
  3. 不激活的专家仅占用显存，完全不参与计算。
  4. MoE 的条件控制是网络内部的可训练路由机制，而非外部规则。

七、程序化路由在其他网络中的应用

🎯 问题

这种程序化的路由控制还在其他类型的网络中使用吗？

回答

按需调用特定子网络的动态路由思想在 NLP、CV、强化学习等领域都非常普遍。

领域 技术示例 核心思想
NLP Switch Transformer, Routing Transformer, COLT5 为每个 token 动态决定使用哪个专家处理
CV V-MoE, MoNE, BiFormer, 动态滤波器 为不同图像区域动态分配计算资源
强化学习 RAPID-RL, REINFORCE 训练门控 根据状态动态选择策略子网络
经典架构 胶囊网络（动态路由，Dynamic Routing）, 自适应 RNN 通过迭代路由协议决定信息路径

核心挑战：离散的”选择”操作通常是不可微的，需要引入负载均衡损失或强化学习来训练门控网络。

📝 知识点小结

  1. 程序化路由思想在 NLP、CV、强化学习中广泛使用。
  2. 所有这些都是”条件计算”——网络内部嵌入可学习的路由控制器。
  3. 核心挑战是离散选择操作不可微，需辅助损失或强化学习。

八、”稀疏”计算模式的理解

🎯 问题

稀疏计算模式这个”稀疏”怎么理解？

回答

在 MoE 语境下，稀疏 = 每次只激活一小部分专家，其余完全不参与计算。体现为路由权重稀疏和计算图稀疏两层含义。

模式 参数参与比例 比喻
稠密 100% 每次开会，所有员工都得发言
稀疏 5%-10% 每次开会，只叫最相关的两三个专家

稀疏的两层含义：

  1. 路由权重稀疏：权重向量经 TopK 后只有 k 个位置非零。
  2. 计算图稀疏：代码只对那 k 个专家调用 forward，不计算其余专家。

📝 知识点小结

  1. 稀疏 = 大部分为零或不参与计算，只有一小部分非零或活跃。
  2. MoE 的稀疏体现在路由权重和计算图两个层面。
  3. 这是 MoE 能用小算力撬动大容量的核心机制。

九、提升大模型性能的其他手段

🎯 问题

除了程序化路由，还有哪些提升大模型性能的手段？

回答

模型压缩

技术 原理 典型收益
量化（Quantization） FP32/FP16 转为 INT8/INT4 体积减 75%，推理加速 2-4 倍
知识蒸馏（Knowledge Distillation） 大模型指导小模型 小模型达大模型 80-90% 性能
剪枝（Pruning） 移除冗余参数/模块 MoE 可剪掉 75% 专家而精度无损

训练与微调

技术 作用
混合精度训练（Mixed Precision Training） FP16 前向 + FP32 梯度累积，加速训练
分布式训练（Distributed Training, ZeRO） 模型状态拆分到多卡，训练超大模型
LoRA（低秩适应） 冻结原模型，仅训练低秩适配器

推理优化

技术 加速比
KV 缓存（Key-Value Cache） 解码速度提升数十倍
PagedAttention（vLLM） 吞吐量提升 2-4 倍
推测解码（Speculative Decoding） 2-3 倍无损加速
FlashAttention 2-4 倍加速，内存降 5-20 倍

架构创新

  • Mamba：基于选择性状态空间模型，线性复杂度
  • 线性注意力：将 softmax 注意力替换为线性核

📝 知识点小结

  1. 模型压缩（量化、蒸馏、剪枝）直接减小模型体积。
  2. 训练微调技术（混合精度、ZeRO、LoRA）提升训练效率。
  3. 推理优化（KV 缓存、PagedAttention、推测解码）显著降低延迟。
  4. 架构创新（Mamba）从根源降低计算复杂度。

十、MoE 的”按需加载”专家策略

🎯 问题

如果每次门控选择后才把对应专家搬进显存，会受到什么影响？

回答

按需加载可以大幅降低显存占用（50%-85%），但会引入推理延迟（增加 47% 至数倍），需要在延迟和显存之间权衡。

维度 变化
显存占用 减少 50%-85%
推理延迟 增加 47% 至数倍（PCIe 传输成为瓶颈）
吞吐量 下降约 30%（优秀系统可控）
模型精度 无损

需要复杂的预取、缓存（LRU）、预测调度策略来优化加载过程。

📝 知识点小结

  1. 按需加载可大幅降低显存门槛（50%-85%），让大模型在消费级 GPU 上运行。
  2. 核心代价是推理延迟增加（47% 至数倍），PCIe 传输是瓶颈。
  3. 模型精度无损，适合消费级硬件推理和边缘计算。

十一、推理延迟 vs. 推理速度

🎯 问题

推理延迟和推理速度是不是两回事？前者来源于上下文编码，后者是前向传播？

回答

推理延迟是端到端时间，受预填充和解码共同影响；推理速度专指解码阶段的生成速率。哪个主导取决于输入/输出长度比例。

术语 英文 定义 度量
推理延迟 Inference Latency 一个请求从输入到输出的端到端总时间 毫秒/秒
推理速度 Throughput 单位时间生成的 token 数（解码阶段） tokens/秒

总延迟 = 预填充时间 + 解码步数 × 单步解码时间

  • 预填充阶段（Prefill Phase）：处理整个 prompt，耗时与输入长度成正比
  • 解码阶段（Decoding Phase）：逐 token 生成，每个 token 一次前向传播

输入短、输出长 → 解码主导延迟；输入长、输出短 → 预填充主导延迟。

📝 知识点小结

  1. 推理延迟不等于推理速度。
  2. 延迟受预填充和解码共同影响，具体哪个主导取决于输入/输出长度比例。
  3. MoE 由于只激活部分专家，解码速度通常比同参数稠密模型快。

十二、KV 缓存的工作原理与位置编码

🎯 问题

KV 缓存是怎么工作的？它和位置编码是什么关系？每生成一个 token 都要完整前向传播吗？RoPE 旋转会转一整圈吗？

回答

KV 缓存存储历史 Key/Value 避免重复计算，将 O(N³) 降至 O(N²)。KV 受位置编码影响，但增量构建无需更新。RoPE 的周期性通过多频率设计和外推技术解决。

KV 缓存（Key-Value Cache）工作原理

KV 缓存的核心思想：避免在生成每个新 token 时重复计算之前已经算过的 Key 和 Value 向量。

有缓存的工作方式：

  1. 每一层 Transformer 维护一个 KV_cache，保存所有历史 token 的 Key 和 Value。
  2. 生成新 token t 时：只计算 Q_t、K_t、V_t，将 K_t、V_t 追加到缓存。
  3. 注意力计算：用 Q_t 与缓存中所有 K 点积，再与所有 V 加权求和。
维度 无 KV 缓存 有 KV 缓存
生成第 L 个 token O(L²) O(L)
总生成 N 个 token O(N³) O(N²)
显存占用 几乎为 0 随序列长度线性增长

重要细节

  • Query 不会被缓存
  • 每层缓存独立
  • 预填充也会生成缓存
  • KV 缓存与模型参数分开存储

KV 缓存与位置编码的关系

KV 实际上是受位置编码（Positional Encoding）影响的。缓存的 K 和 V 已经隐含了该 token 的位置信息。不同位置的同一个 token，其 KV 向量不同（尤其在 RoPE 中，K 会随位置旋转）。

KV 缓存是增量构建的，已有的 KV 条目永远不变。历史 KV 不会因新 token 的到来而更新。

RoPE 旋转位置编码的周期问题

RoPE 的旋转根据 Token 的绝对位置计算。旋转角度会不断增加，高频维度几十到几百 token 就会转一整圈。

多频率时钟系统：

维度对 频率 功能
低频（高 i） 慢 编码模糊的全局绝对位置（”时针”）
高频（低 i） 快 编码精细的局部相对位移（”秒针”）

挑战：频谱损坏（Spectrum Damage）——深层网络中不同频率信号相互”污染”，导致模型在超长序列上性能下降。

解决方案：

  • 增大 base（10000 → 500000），LLaMA 3 采用
  • 线性插值（Linear Interpolation）、NTK-aware scaling、YaRN
  • FoPE（多频率语义，从根本上缓解频谱损坏）

📝 知识点小结

  1. KV 缓存存储历史 Key/Value，避免重复计算，O(N³) → O(N²)。
  2. KV 已含位置信息，且是增量构建无需更新。
  3. RoPE 通过多频率设计让”转圈”成为编码能力的一部分。
  4. 频谱损坏是主要挑战，增大 base、NTK-aware scaling、YaRN、FoPE 是主要手段。

十三、完整知识总结

🎯 问题

总结一下今天聊的全部知识点，再详细点。

回答

本对话覆盖了 MoE 架构、门控机制、专家结构、稀疏计算、推理优化、KV 缓存、RoPE 位置编码等核心概念。

以下为大幅扩充细节的完整知识笔记，保留了原对话中的所有关键点、示例数据、公式直觉和对比表格。

1. 混合专家模型（MoE）核心思想

  • MoE = 稀疏激活的混合专家模型，在 Transformer 的 FFN 层中替换为多个并行的”专家”，每个专家是一个小型 FFN。
  • 总参数量巨大，但每次推理只激活一小部分专家（如 2 个）。
  • 计算稀疏性：用可控的计算成本获得极大的模型容量。
  • 典型模型：Mixtral 8×7B（E=8, k=2），DeepSeek-V2/V3/V4。

2. 门控网络（Router）深入剖析

  • 输入：与专家完全相同的隐藏状态向量 x。
  • 输出：长度为 E 的权重向量。
  • 门控网络不是分类器——它学习的是与专家输出幅值耦合的权重。
  • 联合训练：损失同时影响门控和所有被激活的专家。加入辅助损失（负载平衡损失）防止专家”饿死”。

3. 专家的真实结构

  • 每个专家就是一个标准的 FFN（两个线性层 + 激活函数）。
  • 所有专家结构相同、输入输出维度一致。
  • 专家的唯一输入：该 MoE 层的输入向量 x（门控不提供第二个输入）。

4. 为什么不能随便”删除”或”置零”专家？

  • 数值示例：正常输出 8.841，置零后输出 1.241（暴跌）。
  • 根本原因：门控权重与专家输出幅值深度耦合。
  • 正确方法：静态剪枝（合并冗余专家）或动态加载（按需调换）。

5. 程序化路由在其他网络中的应用

  • NLP：Switch Transformer（单专家），Routing Transformer（稀疏注意力）
  • CV：V-MoE（图像块分配），BiFormer（动态聚焦）
  • RL：RAPID-RL（动态深度），REINFORCE 训练门控
  • 本质：条件计算——网络内嵌入可学习路由控制器

6. 稀疏计算模式

  • 稀疏 = 大部分为零，只有一小部分非零或活跃。
  • MoE 的稀疏：路由权重稀疏 + 计算图稀疏。
  • 稠密模型 100% 参数参与计算；MoE 仅 5%-10%。

7. 提升大模型性能的其他手段

  • 模型压缩：量化、知识蒸馏、剪枝
  • 训练/微调：混合精度训练、ZeRO、LoRA
  • 推理优化：KV 缓存、PagedAttention、推测解码、FlashAttention
  • 架构创新：Mamba（线性序列建模）

8. 推理延迟 vs. 推理速度

  • 推理延迟（Latency）：端到端总时间 = 预填充 + 解码。
  • 推理速度（Throughput）：解码阶段 tokens/秒。
  • 哪个主导取决于输入/输出长度比例。

9. KV 缓存与位置编码

  • KV 缓存避免重复计算，O(N³) → O(N²)。
  • KV 已含位置信息，增量构建无需更新。
  • RoPE 的周期性通过多频率设计和外推技术解决。

📝 知识点小结

  1. MoE 用稀疏激活实现”大容量、低计算”。
  2. 门控网络是动态路由器，与专家联合训练。
  3. 不能随意删除专家，可通过静态剪枝或动态加载实现”瘦身”。
  4. KV 缓存是推理加速的基石，与位置编码协同工作。
  5. RoPE 的周期性既是优点也是挑战，通过增大 base、插值等方法解决。
